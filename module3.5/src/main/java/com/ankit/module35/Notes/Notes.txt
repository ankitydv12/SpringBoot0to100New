/*
===========================================
            FETCH TYPE IN JPA
===========================================
-------------------------------------------
1. WHAT IS FETCH TYPE?
-------------------------------------------
FetchType defines:
"WHEN Hibernate should load related entities
from the database"

Two types:
1. EAGER → load immediately
2. LAZY  → load only when needed

-------------------------------------------
2. FETCHTYPE.EAGER
-------------------------------------------
Definition:
Related entity is loaded immediately along with parent.
Example:
@OneToOne(fetch = FetchType.EAGER)
private Insurance insurance;
-------------------------------------------
BEHAVIOR:
------------------------------------------
When you fetch Patient:

SELECT * FROM patient;
SELECT * FROM insurance WHERE id = ?;

👉 Insurance is loaded automatically
-------------------------------------------
PROBLEM:
-------------------------------------------
- Unnecessary data loading
- Performance issues
- Multiple queries (N+1 problem)
-------------------------------------------
REALITY:
-------------------------------------------
❌ Avoid using EAGER in most cases
-------------------------------------------
3. FETCHTYPE.LAZY
-------------------------------------------
Definition:
Related entity is NOT loaded initially.
It is loaded ONLY when accessed.
Example:
@OneToOne(fetch = FetchType.LAZY)
private Insurance insurance;
-------------------------------------------
BEHAVIOR:
-------------------------------------------
When fetching Patient:
SELECT * FROM patient;
👉 Insurance is NOT loaded
When you call:
patient.getInsurance();
👉 THEN Hibernate runs:
SELECT * FROM insurance WHERE id = ?;
-------------------------------------------
ADVANTAGE:
-------------------------------------------
✔ Better performance
✔ Loads only required data
-------------------------------------------
4. DEFAULT FETCH TYPES (VERY IMPORTANT)
-------------------------------------------
| Relationship       | Default Fetch |
|------------------|--------------|
| @OneToOne         | EAGER        |
| @ManyToOne        | EAGER        |
| @OneToMany        | LAZY         |
| @ManyToMany       | LAZY         |

-------------------------------------------
INTERVIEW TIP:
-------------------------------------------
Always remember:
ToOne → EAGER
ToMany → LAZY
-------------------------------------------
5. LAZY LOADING INTERNALLY
-------------------------------------------
Hibernate uses:
👉 Proxy objects
Instead of actual object, it loads:
Insurance$HibernateProxy
When accessed → actual query fires
-------------------------------------------
6. LAZY INITIALIZATION EXCEPTION
-------------------------------------------
Very common error:
org.hibernate.LazyInitializationException:
could not initialize proxy - no Session
-------------------------------------------
WHEN IT HAPPENS:
-------------------------------------------
1. Transaction is closed
2. You try to access lazy field
Example:
Patient p = repository.findById(1);
session closes
p.getInsurance(); ❌ ERROR
-------------------------------------------
7. HOW TO FIX LAZY ISSUE
-------------------------------------------
✔ Solution 1: Use @Transactional
✔ Solution 2: Use JOIN FETCH in query
✔ Solution 3: Convert to DTO inside transaction
-------------------------------------------
BEST PRACTICE:
-------------------------------------------
Never expose entities directly in controller
-------------------------------------------
8. N+1 QUERY PROBLEM
-------------------------------------------

Example:
    @OneToMany(mappedBy = "patient",cascade = CascadeType.ALL,fetch =  FetchType.EAGER)
    private Set<Appointment> appointment = new HashSet<>(); //inverse side
For each patient:
→ insurance is fetched separately
Result:
1 query for patients + N queries for appointment
-------------------------------------------
SOLUTION:
-------------------------------------------
Use custom query with JOIN FETCH:
SELECT p FROM Patient p Left JOIN FETCH p.appointments --->(this is jpql) or sql query (Select p from Patient p left join on p.id = appointment.id)
-------------------------------------------
9. INDUSTRY BEST PRACTICE
-------------------------------------------
✔ Always use LAZY loading
✔ Fetch data explicitly using queries(i.e custom query)
✔ Avoid EAGER unless absolutely needed


-------------------------------------------
10. YOUR PROJECT CONTEXT
-------------------------------------------

Patient → Insurance

Better to write:

@OneToOne(fetch = FetchType.LAZY, cascade = ALL, orphanRemoval = true)
private Insurance insurance;

-------------------------------------------
WHY?
-------------------------------------------

- Insurance is not always needed
- Saves performance


-------------------------------------------
11. INTERVIEW QUESTIONS
-------------------------------------------

Q1. What is FetchType?
→ Defines when related data is loaded
Q2. Default fetch type of OneToMany?
→ LAZY
Q3. What is LazyInitializationException?
→ Accessing lazy data outside session
Q4. How to solve N+1 problem?
→ JOIN FETCH
Q5. Which fetch type is better?
→ LAZY (in most cases)
-------------------------------------------
12. MEMORY TRICK
-------------------------------------------
EAGER:
"Load everything immediately"
LAZY:
"Load only when needed"
-------------------------------------------
END OF NOTES
-------------------------------------------
